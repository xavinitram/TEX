# BENCH-1 — cumulative benchmark bracketing (v0.20 ↔ v0.28) + the v0.29 gate

*Design/record doc (DOC-6 third layer). Landed with v0.29 "Close the register".*

Eight releases (v0.21–v0.28) each claimed invariant-#7 perf-neutrality on the default ComfyUI
cook path. BENCH-1 is the roadmap's instruction to make the **cumulative** number exist once, in
writing — a v0.20.0-vs-v0.28.0 eight-config compare, a durable baseline JSON, and PM-5's governor
soak in the same idle sitting.

## 1. The v0.29 perf gate (what this release actually needed)

v0.29 is a consolidation release, and its changes are **perf-neutral on the default path by
construction** — verified two ways, independent of the thermally-noisy eight-config bench:

- **The in-suite node-path perf gate holds.** `PR-LP2` (`test_v018_precision.py`) times the real
  `TEXWrangleNode.execute` path at 1024²/2048² and fails below 0.85×. It **passes 3/3 standalone on
  a cool GPU (1.00× / 1.00× / 1.01×)**. (A single contended in-suite reading dipped to 0.15× when the
  bench ran back-to-back after the PM-5 soak's ~GB of disk I/O — a known thermal/contention flake, not
  a code regression; the standalone re-run on a settled GPU is the honest measurement.)
- **Per-item analysis of the default path:**
  - **ENG-4** is an error-path refactor. The only default-path delta is one extra memoized
    `get_cache()` (~100 ns) + `sys.modules`-cached import lookups (~1–2 µs) inside `_compile_or_raise`
    — 3–6 orders of magnitude under a cook. `prepare()` runs once per node execution, not in a loop.
  - **FUS-1b/1c** are gated by the fusion flag and only change fusion *detection/collapse* (queue
    time, not the interactive cook). A single-injection region is byte-identical (schema 1).
  - **SCHED-3 bridge** adds `cancel=get_host_services().cancel_token()`. Off ComfyUI the token is
    `None`; under ComfyUI it is a read-only flag poll at the existing yield points — the `_cancel_check`
    seam is the same one-`is not None`-per-yield cost SCHED-3 already measured in v0.27.
  - **spatial-scalar** changes codegen **only** for a single-channel access/assign on a NON-vector
    base (`@mask.r`, `scalar.r`) — a rare/degenerate pattern; the common `@image.rgb`/`.r` vector
    fast-path is untouched (confirmed: `vec3 @A.r` still fast-paths, bit-exact). count_var declines a
    stencil pattern the parity fuzzer never generates (codegen falls to a correct generic loop).

So v0.29's invariant-#7 gate is **met by the passing PR-LP2 node-path test + this analysis**, not by
the cumulative eight-config bench (which measures the v0.20→v0.28 journey, a separate record).

## 2. The cumulative v0.20 ↔ v0.28 compare (rides the idle-box sitting)

The roadmap's landing rule is explicit: *"eight_config_bench.py --save before the first change of a
release, --compare before tagging … Machine idle, on AC."* The full run is ~2–4 h **per build** (280
compiled subprocesses × cold `torch.compile`), and this laptop's eight-config bench has a **~17%
geomean thermal-noise floor mid-session** (two identical-code runs differ by up to 17% geomean, with
per-program outliers to 14× on a cold GPU). A mid-session run therefore cannot resolve a sub-1%
change and would be misleading. The compare is run **once, on an idle box, on AC** — the durable
JSON is the record, read as a directional trend, not as a `<0.95`-gate off a single noisy run.

### The critical gotcha (the `TEX_Wrangle` symlink)

`benchmarks/run_benchmarks.py` is **byte-identical** v0.20↔v0.28, and both bench scripts import the
package as `TEX_Wrangle` — which is a **symlink** in `custom_nodes/` pointing at the *current* checkout.
So which code gets measured is decided by **where `TEX_Wrangle` resolves, not which worktree's bench
you launch**. A naive `git worktree add` + run-its-bench would silently benchmark v0.28 twice. The
correct recipe points a fresh `TEX_Wrangle` name at the v0.20 checkout:

```bash
# v0.20.0 baseline (85610f3) — run on an idle box, on AC:
git worktree add /c/scratch/bench_v020/checkout v0.20.0
ln -s /c/scratch/bench_v020/checkout /c/scratch/bench_v020/TEX_Wrangle
python /c/scratch/bench_v020/checkout/benchmarks/eight_config_bench.py \
       --save <repo>/benchmarks/results/bench1_v020_baseline.json
git worktree remove /c/scratch/bench_v020/checkout        # when done

# current build (v0.28/v0.29), from the repo (TEX_Wrangle -> main):
python benchmarks/eight_config_bench.py \
       --save benchmarks/results/bench1_v028.json \
       --compare benchmarks/results/bench1_v020_baseline.json
```

To make the baseline JSON **survive the session** in git (both `benchmarks/results/` and `results/`
are `.gitignore`d), mirror the `tests/compat_corpus_goldens.json` precedent: whitelist the one file —
add `!benchmarks/results/bench1_v020_baseline.json` after the `results/` ignore line — so the durable
baseline commits while the rest of `results/` stays ignored. Paste the `--compare` geomean table into
§3 below.

## 4. PM-5 governor soak — DONE

The second half of PM-5 (a many-node graph stays under budget with the CACHE-5 governor) landed as
`test_pm5_governor_soak` (`tests/test_v029_phase1.py`): 50 frames (including a full 4096² result) held
under a tight governor budget across six arbitration cycles, with a flat RSS watermark. The
arbitration is frame-size-agnostic (the existing `test_cache5_governor` proves the spill/restore at a
small resolution); this soak adds the **scale + churn + flat-watermark** dimension. *(PM-5's first
half — an 8K `gauss_blur` via halo tiling — shipped and is gated in v0.27's `test_roi5_halo_tiling`.)*

## 5. Canonical suite count — reconciled

The suite total is **2232-class**; the ±1 the audit saw is a single flaky sub-test, not a code change:

- `tests/helpers.py` counts `total = passed + failed` (skips excluded).
- `test_integration.test_examples` compiles every `examples/*.tex` under a 30 s per-program timeout in
  a thread pool. `examples/merge.tex` (a fused multi-blend STOCK exemplar) has an expensive
  `torch.compile` that can exceed 30 s **under whole-suite GPU contention** — on timeout it emits a
  *named* `r.skip("example compiled: merge", …)` rather than a silent drop.
- **merge compiles within 30 s → counted → total 2232; merge times out → skipped → total 2231.** That
  is the entire ±1. It is load-dependent, not a nondeterministic race, and not a code bug.
- The one **persistent** failure across every recent release is **S-4** (`tex_validate_hw` printing to
  a cp1252 console under the test harness — an environment/console-encoding gap on this dev box, not a
  code defect; identical on committed v0.27.0/v0.28.0).

With the v0.29 test additions (ENG-4 recut, SCHED-3 bridge ×2, count_var, spatial-scalar, FUS-1b,
PM-5 soak), the suite measures **~2262–2264** (the exact total tracks the merge flake and the CUDA
device count); the only genuine failure remains S-4.
